Spring Data JPA

Java obyektlari va relatsion bazalar o'rtasidagi sehrli ko'prik. ORM, Entity, Repository, Relationships va kuchli query'lar.

Dars maqsadi: ORM tushunchasi, Entity Lifecycle, Persistence Context, Repositories, Relationships (OneToMany, ManyToMany), JPQL, N+1 Problem va Transactions'ni chuqur o'rganamiz.

ORM nima va nima uchun?

ORM (Object-Relational Mapping) — Java obyektlari va relatsion DB jadval'lari o'rtasida avtomatik moslik o'rnatuvchi texnologiya.

Muammo: Java obyektlar bilan, DB jadval bilan ishlaydi. Ular orasida "impedance mismatch" bor.

Yechim: ORM — har bir class'ni jadval bilan, field'ni ustun bilan bog'laydi. SQL yozish o'rniga, obyektlar bilan ishlaysiz.

// ORM SIZ — har SQL'ni qo'lda yozish
String sql = "SELECT id, name, email FROM users WHERE id = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setLong(1, 42);
ResultSet rs = ps.executeQuery();
User user = null;
if (rs.next()) {
    user = new User();
    user.setId(rs.getLong("id"));
    user.setName(rs.getString("name"));
    user.setEmail(rs.getString("email"));
}
rs.close(); ps.close();
// ORM bilan — bir qator!
User user = userRepository.findById(42L).orElseThrow();

Entity Lifecycle — 4 ta holat

JPA'da har bir obyekt quyidagi 4 ta holatdan birida bo'ladi:

┌─────────────┐
│  TRANSIENT  │  new User()
│ (yangi obyekt) │  DB'da yo'q, JPA bilmaydi
└──────┬──────┘
       │ persist() yoki save()
       ↓
┌─────────────┐
│  PERSISTENT │  Persistence Context'da, DB'da bor
│  (managed)  │  Field o'zgartirilsa — DB ga avtomatik yoziladi!
└──┬──────┬───┘
   │      │ detach() / close()
   │      ↓
   │ ┌─────────────┐
   │ │  DETACHED   │  PC'dan chiqdi, DB'da hali ham bor
   │ │  (uzilgan)  │  O'zgarishlar saqlanmaydi
   │ └─────────────┘
   │
   │ remove() / delete()
   ↓
┌─────────────┐
│   REMOVED   │  O'chirilishi belgilangan
│             │  Transaction commit'da DB'dan o'chadi
└─────────────┘
Dirty checking — PERSISTENT holatda obyektning field'lari o'zgartirilsa, JPA buni sezadi va transaction oxirida UPDATE SQL'ni avtomatik chaqiradi. save() chaqirish shart emas!

Persistence Context — 1st level cache

Persistence Context (PC) — JPA'ning xotirada saqlanadigan entity'lar to'plami. EntityManager bilan birga yashaydi (odatda transaction davomida).

Xususiyatlari:

1️⃣ 1st level cache — bir transaction ichida bir xil ID bo'yicha murojaat — DB ga faqat 1 marta boradi

2️⃣ Dirty checking — entity field'lari o'zgarishini kuzatib boradi

3️⃣ Lazy loading — kerakli vaqtda DB'dan qo'shimcha ma'lumot oladi

4️⃣ Write-behind — INSERT/UPDATE/DELETE'ni transaction oxirida birga jo'natadi

@Transactional
public void example() {
    User u1 = userRepo.findById(42L).orElseThrow();  // SQL: SELECT
    User u2 = userRepo.findById(42L).orElseThrow();  // SQL chaqirilmaydi! Cache'dan

    System.out.println(u1 == u2);  // true — bir xil obyekt!

    u1.setName("New Name");        // save() chaqirish SHART EMAS
    // Transaction oxirida — SQL: UPDATE
}

JpaRepository va Derived Queries

@Repository
public interface UserRepository extends JpaRepository<User, Long> {

    // 1. JpaRepository tayyor metodlar bilan keladi:
    // findById(id), findAll(), save(entity), delete(entity), count(), ...

    // 2. DERIVED QUERIES — metod nomidan SQL generate qilinadi
    Optional<User> findByEmail(String email);
    List<User> findByNameContainingIgnoreCase(String name);
    List<User> findByAgeGreaterThan(int age);
    List<User> findByStatusAndAgeBetween(UserStatus s, int min, int max);
    List<User> findTop10ByOrderByCreatedAtDesc();
    boolean existsByEmail(String email);
    long countByStatus(UserStatus status);
    int deleteByStatus(UserStatus status);

    // 3. PROJECTIONS — faqat ba'zi field'lar

        Long getId();
        String getName();
    }
    List<UserSummary> findAllProjectedBy();
}
Konvensiya: findBy, countBy, existsBy, deleteBy + field nomi + operator. Operator'lar: Containing, StartingWith, GreaterThan, Between, In, OrderBy va h.k.

@Query — JPQL va Native

@Repository
public interface UserRepository extends JpaRepository<User, Long> {

    // JPQL — Entity nomlari ishlatiladi (DB jadvallari emas!)
    @Query("SELECT u FROM User u WHERE u.email = :email")
    Optional<User> findByEmailQuery(@Param("email") String email);

    @Query("SELECT u FROM User u WHERE u.age > :age ORDER BY u.name")
    List<User> findOlderThan(@Param("age") int age);

    // UPDATE/DELETE uchun @Modifying kerak
    @Modifying
    @Query("UPDATE User u SET u.status = :status WHERE u.lastLogin < :before")
    int markInactive(@Param("status") UserStatus s, @Param("before") LocalDateTime before);

    // Native SQL — DB-specific imkoniyatlar uchun
    @Query(value = "SELECT * FROM users WHERE created_at > NOW() - INTERVAL '7 days'",
           nativeQuery = true)
    List<User> findRecentNative();

    // Constructor DTO projection
    @Query("""
        SELECT new com.example.UserDto(u.id, u.name, u.email)
        FROM User u
        WHERE u.status = 'ACTIVE'
    """)
    List<UserDto> findActiveDtos();
}
JPQL vs SQL farqi: JPQL Entity klass va field nomlarini ishlatadi (case-sensitive). Native SQL — jadval va ustun nomlarini.

Specification — dynamic query

Frontend turli filter'lar yuborishi mumkin. Hech qaysi field majburiy emas. Specification API bilan dinamik so'rov. 

public class ProductSpec {

    public static Specification<Product> hasCategory(String category) {
        return (root, query, cb) ->
            category == null ? null : cb.equal(root.get("category"), category);
    }

    public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (min == null && max == null) return null;
            if (min == null) return cb.lessThanOrEqualTo(root.get("price"), max);
            if (max == null) return cb.greaterThanOrEqualTo(root.get("price"), min);
            return cb.between(root.get("price"), min, max);
        };
    }

    public static Specification<Product> nameContains(String name) {
        return (root, query, cb) ->
            name == null ? null : cb.like(cb.lower(root.get("name")), "%" + name.toLowerCase() + "%");
    }
}

// Repository
public interface ProductRepository extends JpaRepository<Product, Long>,
                                            JpaSpecificationExecutor<Product> { }

// Foydalanish — har xil filter'larni birlashtirish
Page<Product> result = productRepo.findAll(
    Specification.where(ProductSpec.hasCategory(filter.getCategory()))
        .and(ProductSpec.priceBetween(filter.getMin(), filter.getMax()))
        .and(ProductSpec.nameContains(filter.getName())),
    PageRequest.of(0, 20)
);

Relationships — 4 ta turi

TuriMisolAnnotation
@OneToOneUser ↔ Profile@OneToOne
@OneToManyUser → ko'p Order@OneToMany(mappedBy)
@ManyToOneOrder → bitta User@ManyToOne + @JoinColumn
@ManyToManyUser ↔ Role@ManyToMany + @JoinTable

Owner side va Inverse side:

Owner — Foreign key'ga ega tomon. Database'da bog'liqlikni saqlaydi.

Inverse — mappedBy orqali ko'rsatiladi. Owner side'ga "qarab turadi".

Bidirectional relationship'da: owner side — ko'pincha @ManyToOne.

@OneToMany / @ManyToOne

@Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;

    // INVERSE side — DB'da column yo'q, faqat navigatsiya uchun
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Order> orders = new ArrayList<>();

    // Helper metod — bog'lashni to'g'ri saqlash uchun
    public void addOrder(Order order) {
        orders.add(order);
        order.setUser(this);
    }
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private BigDecimal total;

    // OWNER side — FK shu yerda
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User user;
}

// DB:
// users (id, name)
// orders (id, total, user_id ← FK)
Diqqat: @ManyToOne default EAGER yuklash bilan keladi — bu yomon! Har doim fetch = LAZY qo'ying.

@ManyToMany — User ↔ Role

@Entity
public class User {

@Entity

Cascade Types

Cascade — parent entity'da bajarilgan amalni child'larga "tarqatish".

CascadeTavsif
PERSISTsave() parent → child ham save
MERGEupdate() parent → child ham update
REMOVEdelete() parent → child ham delete
REFRESHDB'dan qayta o'qish — child'lar ham
DETACHPC'dan chiqarish — child'lar ham
ALLHammasini birga
 
@OneToMany(mappedBy = "user",
           cascade = CascadeType.ALL,
           orphanRemoval = true)    // child'ni list'dan olib tashlasak — DB'dan ham

Lazy vs Eager Loading

Lazy — relation'ni kerakli vaqtda DB'dan oladi. Default'da @OneToMany va @ManyToMany uchun.

Eager — relation'ni entity'ni o'qish bilan birga DB'dan oladi. Default'da @ManyToOne va @OneToOne uchun (LEKIN BU YOMON!).

// LAZY — child'lar darhol yuklanmaydi
@Entity
public class User {
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;
}

@Transactional
public void example() {
    User user = userRepo.findById(42L).orElseThrow();
    // SQL: SELECT * FROM users WHERE id = 42
    // Bu yerda orders YUKLANMAGAN — proxy obyekt

    System.out.println(user.getOrders().size());
    // SQL: SELECT * FROM orders WHERE user_id = 42
    // Aniq foydalanilganda yuklanadi
}
LazyInitializationException — transaction tugagandan keyin lazy field'ga murojaat qilsangiz! Hamma kerakli ma'lumotni transaction ichida oling yoki JOIN FETCH ishlating.

N+1 Query Problem — ENG MUHIM!

N+1 Problem — JPA'da eng ko'p uchraydigan performance muammosi.

1 ta query parent'larni oladi → keyin har bir parent uchun 1 ta query child'larni oladi = N+1 ta query!

100 ta user uchun — 101 ta SQL!

// YOMON KOD — N+1
@Transactional
public void badExample() {
    List<User> users = userRepo.findAll();
    // SQL #1: SELECT * FROM users    (100 ta user)

    for (User u : users) {
        System.out.println(u.getOrders().size());
        // SQL #2..#101: SELECT * FROM orders WHERE user_id = ?
        // 100 ta SQL!
    }
}

// LOG'da ko'rinishi:
// Hibernate: SELECT u.* FROM users u
// Hibernate: SELECT o.* FROM orders o WHERE o.user_id = 1
// Hibernate: SELECT o.* FROM orders o WHERE o.user_id = 2
// ... (98 marta yana) ...
// Hibernate: SELECT o.* FROM orders o WHERE o.user_id = 100
Yechim: JOIN FETCH, @EntityGraph, yoki @BatchSize.

JOIN FETCH va @EntityGraph — N+1 yechimi

// YECHIM 1: JOIN FETCH (JPQL)
@Query("SELECT DISTINCT u FROM User u LEFT JOIN FETCH u.orders")
List<User> findAllWithOrders();
// SQL: SELECT * FROM users u
//      LEFT JOIN orders o ON o.user_id = u.id
// Bitta SQL — hamma narsa keladi!

// YECHIM 2: @EntityGraph
@EntityGraph(attributePaths = {"orders"})
@Query("SELECT u FROM User u")
List<User> findAllGraph();

// Yoki tayyor metodga
@EntityGraph(attributePaths = {"orders", "profile"})
List<User> findByStatus(UserStatus status);

// YECHIM 3: @BatchSize — pessimistic
@Entity
public class User {
    @OneToMany(mappedBy = "user", fetch = LAZY)
    @BatchSize(size = 50)
    private List<Order> orders;
}
// 100 ta user → 100 + (100/50) = 102 ta SQL emas, 1+2=3 ta!
JOIN FETCH'da DISTINCT kerak bo'lishi mumkin — chunki LEFT JOIN parent'larni dublikat qilib qaytaradi.

@Transactional — chuqurroq nazariya

@Transactional — metod boshlanishida transaction ochadi, tugashida commit yoki rollback qiladi.

Propagation — joriy transaction bo'lsa nima qilish kerak (REQUIRED — qo'shilish, REQUIRES_NEW — yangi ochish, SUPPORTS — bor bo'lsa qo'shilish).

Isolation — boshqa transaction'larga qanday munosabat (READ_COMMITTED, REPEATABLE_READ va h.k.).

RollbackFor — qaysi exception'larda rollback qilish (default: faqat RuntimeException!).

@Service
public class OrderService {

    @Transactional
    public Order create(OrderRequest req) {
        // ↑ default: REQUIRED + READ_COMMITTED + faqat RuntimeException uchun rollback
    }

    @Transactional(readOnly = true)
    public List<Order> findAll() {
        // read-only — optimizatsiya: Hibernate dirty checking'ni o'chiradi
    }

    @Transactional(
        propagation = Propagation.REQUIRES_NEW,
        isolation = Isolation.SERIALIZABLE,
        rollbackFor = Exception.class,
        timeout = 10
    )
    public void criticalPayment() { /* ... */ }
}
Eslatma: @Transactional faqat boshqa bean'dan chaqirilganda ishlaydi (proxy)! Bir xil klass ichida this.method() chaqirsangiz — transaction boshlanmaydi.

Soft Delete va @SQLDelete

Ko'p loyihalarda haqiqiy DELETE o'rniga "soft delete" ishlatiladi — flag o'rnatiladi, ma'lumot saqlanadi. 

@Entity
@SQLDelete(sql = "UPDATE users SET deleted = true WHERE id = ?")
@Where(clause = "deleted = false")
public class User {
    @Id @GeneratedValue
    private Long id;

    @Column(nullable = false)
    private boolean deleted = false;

    private LocalDateTime deletedAt;
    private String deletedBy;
}

// Foydalanish
userRepo.delete(user);
// SQL: UPDATE users SET deleted = true WHERE id = 42
// User DB'da qoladi, lekin findById qaytarmaydi

userRepo.findAll();
// SQL: SELECT * FROM users WHERE deleted = false (avtomatik)

// Hard delete uchun alohida native query
@Modifying
@Query(value = "DELETE FROM users WHERE id = :id", nativeQuery = true)
void hardDelete(@Param("id") Long id);

10 ta murakkab vazifa

Quyidagi vazifalarni Spring Data JPA bilan bajaring:

  • 1. E-commerce data model. User, Product, Category, Order, OrderItem, Cart — to'liq bog'lanishlar bilan.
  • 2. Multi-level JOIN FETCH. Post → author → comments → comment-author hammasini bitta query'da. Pagination ham.
  • 3. Specification API. Frontend 10+ filter yuborishi mumkin. Hech qaysi majburiy emas. Dynamic.
  • 4. Audit Log. AbstractAuditable + custom AuditorAware (Spring Security'dagi user'ni olish).
  • 5. Soft Delete + Restore. @SQLDelete + @Where + native query bilan tiklash.
  • 6. Bulk Operations. 10000 ta yozuvni saqlash (saveAll). JDBC batch'ni yoqish.
  • 7. Native Query + DTO. Complex SQL (window functions, CTE) → DTO projection.
  • 8. Versioning (@Version). Optimistic Locking bilan concurrent update muammosini hal qilish.
  • 9. Transactional Outbox Pattern. Event'larni jadval orqali jo'natish — distributed system'lar uchun.
  • 10. Performance optimization. Slow query'ni profile qilish, EXPLAIN ANALYZE, index'lar qo'shish.
Maslahat: spring.jpa.show-sql=true + logging.level.org.hibernate.SQL=DEBUG bilan SQL'larni ko'rib turing — N+1 va ortiqcha so'rovlarni darhol sezasiz.
