SQL va Database

Ma'lumotlarni saqlash, qidirish va boshqarish uchun dunyodagi eng muhim til. Backend uchun majburiy.

Dars maqsadi: Database asoslari, SQL CRUD, JOIN, GROUP BY, Indexes, Transactions (ACID), Isolation levels, JDBC va SQL injection xavfsizligini batafsil o'rganamiz.

Database nima?

Database (ma'lumotlar bazasi) — tartibli, qidirish oson, ko'p foydalanuvchili ma'lumotlar to'plami.

Nima uchun fayl emas? Fayllarda: parallel kirish qiyin, qidirish sekin, tartib yo'q, xavfsizlik yo'q.

DBMS (Database Management System) — bu muammolarni hal qiluvchi tizim: PostgreSQL, MySQL, Oracle, SQLite va h.k.

TuriXususiyatMisol
RDBMSJadval-ustun, SQL bilan, ACIDPostgreSQL, MySQL, Oracle
NoSQLDocument, key-value, graphMongoDB, Redis, Neo4j
NewSQLSQL + ko'lamlashCockroachDB, TiDB
Bu dars asosan RDBMS ga bag'ishlangan. SQL — RDBMS bilan ishlash tili.

Asosiy atamalar

Table

Jadval — ustun va qatorlardan iborat ma'lumot tuzilmasi. Excel jadvali kabi.

Row

Qator (record, tuple) — bitta yozuv. Masalan: bitta foydalanuvchi.

Column

Ustun (field) — ma'lum bir atribut. Masalan: name, age, email.

Primary Key

Har bir qator uchun noyob identifikator. Odatda id.

Foreign Key

Boshqa jadvaldagi qatorga havola. Jadvallar o'rtasidagi bog'lanish.

Index

Tezroq qidirish uchun yordamchi struktura. Kitobning indeksi kabi.

SQL turlari

SQL — Structured Query Language. 1974-yilda IBM yaratgan. ANSI standart bo'lib, har bir DBMS o'z dialekti bilan ishlaydi.

Declarative til: "Nima qilish kerakligi"ni aytasiz, "qanday qilish"ni DBMS o'zi hal qiladi.

TuriMaqsadiMisol

CREATE TABLE — jadval yaratish

CREATE TABLE users (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    email       VARCHAR(255) UNIQUE NOT NULL,
    age         INTEGER CHECK (age >= 0),
    balance     DECIMAL(10, 2) DEFAULT 0,
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Foreign key bilan
CREATE TABLE orders (
    id          SERIAL PRIMARY KEY,
    user_id     INTEGER REFERENCES users(id) ON DELETE CASCADE,
    total       DECIMAL(10, 2),
    created_at  TIMESTAMP DEFAULT NOW()
);

WHERE — filtrlash

-- Oddiy shart
SELECT * FROM users WHERE age > 25;

-- AND, OR
SELECT * FROM users WHERE age > 18 AND is_active = TRUE;
SELECT * FROM users WHERE name = 'Akobir' OR name = 'Ali';

-- IN — ro'yxatdan
SELECT * FROM users WHERE id IN (1, 3, 5);

-- BETWEEN — oraliq
SELECT * FROM users WHERE age BETWEEN 20 AND 30;

-- LIKE — naqsh bo'yicha (% har qanday, _ bitta belgi)
SELECT * FROM users WHERE name LIKE 'A%';     -- A bilan boshlanadi
SELECT * FROM users WHERE email LIKE '%@gmail.com';
SELECT * FROM users WHERE name LIKE '_li';    -- 3 harf, 2-si 'l'

-- NULL tekshirish
SELECT * FROM users WHERE phone IS NULL;
SELECT * FROM users WHERE phone IS NOT NULL;

-- NOT
SELECT * FROM users WHERE NOT (age < 18);

ORDER BY, LIMIT, OFFSET

-- Saralash (default ASC)
SELECT * FROM users ORDER BY age;
SELECT * FROM users ORDER BY age DESC;     -- kamayish

-- Bir nechta ustun bo'yicha
SELECT * FROM users
ORDER BY age DESC, name ASC;

-- Cheklash (paginate uchun)
SELECT * FROM users ORDER BY id LIMIT 10;          -- 10 ta
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 20;-- 21-30

-- Pagination (page=3, size=10)
SELECT * FROM users
ORDER BY id
LIMIT 10 OFFSET 20;

-- DISTINCT — takrorlanmaslar
SELECT DISTINCT city FROM users;
LIMIT va OFFSET — pagination uchun. OFFSET katta bo'lsa sekinlashadi (cursor-based pagination yaxshiroq).

UPDATE va DELETE

-- UPDATE — o'zgartirish
UPDATE users
SET balance = 1000, is_active = TRUE
WHERE id = 5;

-- Bir nechta qator
UPDATE users
SET age = age + 1
WHERE age < 30;

-- DELETE — o'chirish
DELETE FROM users WHERE id = 5;

-- Shartli o'chirish
DELETE FROM users WHERE is_active = FALSE;

-- HAMMA SINI o'chirish (ehtiyot bo'ling!)
DELETE FROM users;
-- yoki
TRUNCATE TABLE users;  -- tezroq, lekin rollback yo'q
OGOHLANTIRISH: UPDATE yoki DELETEni WHERE'siz yozsangiz — butun jadvalga ta'sir qiladi! Har doim avval SELECT bilan tekshiring.

Faollar ro'yxati

users jadvalidan 18+ yoshli va aktiv foydalanuvchilarni toping. Faqat name va email ustunlarini chiqaring, ism alifbo bo'yicha. 

SELECT name, email
FROM users
WHERE age >= 18 AND is_active = TRUE
ORDER BY name ASC;
⚡ Masala 2 Easy

Email domen bo'yicha qidirish

Faqat @gmail.com domenidan foydalanuvchilarni toping va eng katta yoshlilarni birinchi qiling. Faqat 10 tasini qaytaring. 

SELECT *
FROM users
WHERE email LIKE '%@gmail.com'
ORDER BY age DESC
LIMIT 10;
⚡ Masala 3 Medium

Bulk update — balansga foiz qo'shish

Barcha aktiv foydalanuvchilarning balansiga 5% foiz qo'shing, lekin faqat balans 100 dan katta bo'lganlar uchun. 

UPDATE users
SET balance = balance * 1.05
WHERE is_active = TRUE
  AND balance > 100;

-- Yangilangan qatorlar sonini tekshirish (PostgreSQL)
SELECT COUNT(*) FROM users
WHERE is_active = TRUE AND balance > 100;
⚡ Masala 4 Hard

Subquery bilan top-N

Har bir shahardan eng katta yoshli foydalanuvchini toping (subquery yoki window function ishlatib). 

-- Yo'l 1: subquery
SELECT u.*
FROM users u
WHERE age = (
    SELECT MAX(age) FROM users
    WHERE city = u.city
);

-- Yo'l 2: window function (zamonaviy, tezroq)
SELECT *
FROM (
    SELECT u.*,
           ROW_NUMBER() OVER (PARTITION BY city ORDER BY age DESC) AS rn
    FROM users u
) ranked
WHERE rn = 1;

JOIN — jadvallarni birlashtirish

Nima uchun JOIN? Ma'lumot bir jadvalga sig'masligi mumkin. Masalan: foydalanuvchilar va ularning buyurtmalari.

Foreign Key orqali jadvallar bog'lanadi. JOIN — ularni birgalikda ko'rsatish.

JOIN Cartesian product dan boshlanadi (har bir qator har bir qator bilan) va keyin shart bo'yicha filtrlanadi.

-- 2 ta jadval bor:
-- users(id, name)
-- orders(id, user_id, total)

-- INNER JOIN — ikkalasida ham mavjud bo'lganlar
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

JOIN turlari

-- INNER JOIN — faqat moslar
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN — chap (users) qoladi, o'ng yo'q bo'lsa NULL
SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
-- Buyurtmasi yo'q userlar ham chiqadi (total = NULL)

-- RIGHT JOIN — aksincha
SELECT u.name, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

-- FULL OUTER JOIN — ikkala tomondan ham hammasi
SELECT u.name, o.total
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;

-- SELF JOIN — bir jadval o'zi bilan
SELECT e1.name AS employee, e2.name AS manager
FROM employees e1
LEFT JOIN employees e2 ON e1.manager_id = e2.id;

-- Bir nechta JOIN
SELECT u.name, o.total, p.name AS product
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN products p ON o.product_id = p.id;
⚡ Masala Medium

Buyurtmasiz foydalanuvchilar

Bironta ham buyurtma qilmagan foydalanuvchilarni toping (LEFT JOIN + IS NULL trick).

Misol
users: [Ali, Vali, Sardor, Akobir]
orders: [Vali → buyurtma, Akobir → buyurtma]
→ Natija: Ali, Sardor
Maslahat: LEFT JOIN qiladigan har bir foydalanuvchi qatorida buyurtma yo'q bo'lsa, orders.id IS NULL bo'ladi. 
SELECT u.id, u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

-- Yoki NOT EXISTS bilan (ko'pincha tezroq)
SELECT u.id, u.name
FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);

Aggregate funksiyalar

Aggregate — bir necha qatorni bitta natijaga aylantiradi. 

SELECT COUNT(*)        FROM users;        -- jami qator soni
SELECT COUNT(email)    FROM users;        -- NULL emas emaillar
SELECT COUNT(DISTINCT city) FROM users;   -- noyob shaharlar

SELECT SUM(balance)    FROM users;        -- jami balans
SELECT AVG(age)        FROM users;        -- o'rta yosh
SELECT MIN(age)        FROM users;        -- eng yosh
SELECT MAX(balance)    FROM users;        -- eng katta balans

-- Bitta query'da bir nechtasi
SELECT
    COUNT(*)   AS total_users,
    AVG(age)   AS avg_age,
    MAX(balance) AS richest,
    MIN(balance) AS poorest
FROM users
WHERE is_active = TRUE;
COUNT(*) hamma qatorni sanaydi (NULL ham). COUNT(column) faqat NULL bo'lmaganini sanaydi.

GROUP BY va HAVING

GROUP BY — qatorlarni guruhlarga ajratadi, aggregate'larni har bir guruh uchun alohida hisoblaydi.

HAVING vs WHERE: WHERE guruhlashdan oldin, HAVING keyin filtrlaydi.

-- Har bir shahar bo'yicha foydalanuvchilar soni
SELECT city, COUNT(*) AS users_count
FROM users
GROUP BY city;

-- Saralash bilan
SELECT city, COUNT(*) AS cnt
FROM users
GROUP BY city
ORDER BY cnt DESC;

-- HAVING — guruh ustida filter
SELECT city, COUNT(*) AS cnt
FROM users
GROUP BY city
HAVING COUNT(*) >= 10;   -- faqat 10+ foydalanuvchili shaharlar

-- WHERE + GROUP BY + HAVING — to'liq pipeline
SELECT city, AVG(age) AS avg_age
FROM users
WHERE is_active = TRUE       -- 1. Avval qatorni filtrlash
GROUP BY city                 -- 2. Keyin guruhlash
HAVING AVG(age) > 25         -- 3. Guruhni filtrlash
ORDER BY avg_age DESC;        -- 4. Saralash
⚡ Masala 1 Easy

Buyurtmalar soni va summasi

Har bir foydalanuvchi uchun buyurtmalar soni va jami summasini chiqaring. Eng ko'p sarflaganlar birinchi. 

SELECT
    u.id,
    u.name,
    COUNT(o.id) AS orders_count,
    COALESCE(SUM(o.total), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
ORDER BY total_spent DESC;
⚡ Masala 2 Easy

Eng ko'p sotilgan tovarlar (Top 5)

order_items(order_id, product_id, quantity) jadvalidan eng ko'p sotilgan 5 ta mahsulotni toping. 

SELECT
    p.id,
    p.name,
    SUM(oi.quantity) AS total_sold
FROM products p
JOIN order_items oi ON p.id = oi.product_id
GROUP BY p.id, p.name
ORDER BY total_sold DESC
LIMIT 5;
⚡ Masala 3 Medium

Oylik daromad

orders(created_at, total) jadvalidan har oy bo'yicha jami daromadni chiqaring. Faqat 2025-yil uchun. 

SELECT
    EXTRACT(YEAR FROM created_at) AS year,
    EXTRACT(MONTH FROM created_at) AS month,
    COUNT(*) AS orders_count,
    SUM(total) AS revenue
FROM orders
WHERE EXTRACT(YEAR FROM created_at) = 2025
GROUP BY year, month
ORDER BY year, month;

-- Yoki TO_CHAR bilan
SELECT
    TO_CHAR(created_at, 'YYYY-MM') AS month,
    SUM(total) AS revenue
FROM orders
WHERE created_at >= '2025-01-01'
  AND created_at < '2026-01-01'
GROUP BY TO_CHAR(created_at, 'YYYY-MM')
ORDER BY month;
⚡ Masala 4 Hard

Loyal mijozlar (HAVING + JOIN)

Kamida 5 ta buyurtma qilgan va jami summasi 1 mln'dan ko'p bo'lgan foydalanuvchilarni toping. Har birining o'rta buyurtmasini ham chiqaring. 

SELECT
    u.id,
    u.name,
    COUNT(o.id) AS orders_count,
    SUM(o.total) AS total_spent,
    AVG(o.total) AS avg_order
FROM users u
JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name
HAVING COUNT(o.id) >= 5
   AND SUM(o.total) > 1000000
ORDER BY total_spent DESC;

Indexes — tezlikning siri

Index nima? Jadvaldagi qatorlarga tezroq murojaat qilish uchun yordamchi struktura. Kitobning oxiridagi indeks kabi.

Eng ko'p ishlatiladigan tur — B-Tree (balanced tree). Saralangan ko'p shoxli daraxt. Qidirish O(log n).

Lekin: indeks xotira egallaydi va INSERT/UPDATE/DELETE'ni sekinlashtiradi (chunki indeks ham yangilanishi kerak).

-- Index yaratish
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_age ON users(age);

-- Composite (bir nechta ustun)
CREATE INDEX idx_users_city_age ON users(city, age);

-- Unique index
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);

-- Partial index — faqat shartga mos qatorlar uchun
CREATE INDEX idx_active_users ON users(email) WHERE is_active = TRUE;

-- Index'ni o'chirish
DROP INDEX idx_users_email;

-- Index ishlatildi mi? EXPLAIN bilan ko'rish
EXPLAIN SELECT * FROM users WHERE email = 'x@y.com';
Qoida: Faqat tez-tez qidiriladigan ustunlarga index qo'ying. Har bir indeks — qo'shimcha xarajat.

Transactions — ACID

Tranzaksiya — bir nechta SQL amallari bitta blok sifatida bajariladi. Yoki hammasi, yoki hech qaysi biri.

ACID — 4 ta kafolat:

Atomicity — Atomiklik. Hamma yoki hech qaysi.

Consistency — Mosligi. Tranzaksiyadan oldin va keyin DB to'g'ri holatda.

Isolation — Izolyatsiya. Bir vaqtdagi tranzaksiyalar bir-biriga xalal bermaydi.

Durability — Doimiylik. COMMIT dan keyin ma'lumot saqlanadi (hatto crash bo'lsa ham).

-- Bank o'tkazmasi — klassik misol
BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE id = 1;
    UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- Xato bo'lsa qaytarish
BEGIN;
    UPDATE accounts SET balance = balance - 100 WHERE id = 1;
    -- Bu yerda xato bo'lsa
ROLLBACK;  -- hamma narsa qaytariladi

-- SAVEPOINT — qisman rollback
BEGIN;
    INSERT INTO users (name) VALUES ('A');
    SAVEPOINT sp1;
    INSERT INTO users (name) VALUES ('B');
    ROLLBACK TO sp1;     -- B yo'qoladi, A qoladi
COMMIT;

Isolation Levels

Tranzaksiyalar bir vaqtda ishlaganda muammolar paydo bo'lishi mumkin:

Dirty Read: Boshqa tranzaksiyaning commit'lanmagan ma'lumotini o'qish

Non-repeatable Read: Bir tranzaksiya ichida ikki marta o'qiganda turli natija

Phantom Read: Bir tranzaksiya ichida ikki marta sanaganda turli qator soni

Isolation LevelDirtyNon-repeatPhantom
READ UNCOMMITTED❌❌❌
READ COMMITTED (default ko'p DBMS)✅❌❌
REPEATABLE READ✅✅❌
SERIALIZABLE (eng qattiq)✅✅✅
 
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- amallar
COMMIT;
⚡ Masala 1 Easy

Bank o'tkazmasi (Transaction)

accounts(id, balance) jadvalida 1-account'dan 2-account'ga 500 so'm o'tkazing. Yetarli balans bo'lmasa rollback qiling. 

BEGIN;

-- Balansni tekshirish
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- FOR UPDATE — qatorni qulflaydi (boshqa transaction kuta turadi)

-- Pul yechish
UPDATE accounts SET balance = balance - 500
WHERE id = 1 AND balance >= 500;

-- Pul qo'shish
UPDATE accounts SET balance = balance + 500 WHERE id = 2;

-- Tekshirish: birinchi UPDATE muvaffaqiyatli bo'ldi mi?
-- Agar yo'q (balans yetmadi) — ROLLBACK
-- Agar ha — COMMIT

COMMIT;
⚡ Masala 2 Medium

Index'ni qaerga qo'yish kerak?

Quyidagi query har kuni 1000 marta ishlaydi. Qaysi ustun(lar)ga index qo'yish kerak? 

SELECT *
FROM orders
WHERE user_id = ?
  AND status = 'PENDING'
  AND created_at >= '2025-01-01'
ORDER BY created_at DESC
LIMIT 20;

-- Composite index — eng samarali
CREATE INDEX idx_orders_user_status_date
ON orders(user_id, status, created_at DESC);

-- WHERE user_id = ?

⚡ Masala 3 Medium

Order yaratish (multi-table transaction)

Buyurtma yaratish: orders jadvaliga yangi qator qo'shing va order_items jadvaliga 3 ta tovar qo'shing. Birortasi xato bo'lsa — hamma narsa rollback. 

BEGIN;

INSERT INTO orders (user_id, total)
VALUES (123, 500)
RETURNING id INTO @order_id;  -- PostgreSQL: WITH bilan

INSERT INTO order_items (order_id, product_id, quantity, price) VALUES
    (@order_id, 1, 2, 100),
    (@order_id, 2, 1, 200),
    (@order_id, 3, 1, 100);

-- Stock'ni kamaytirish
UPDATE products SET stock = stock - 2 WHERE id = 1;
UPDATE products SET stock = stock - 1 WHERE id = 2;
UPDATE products SET stock = stock - 1 WHERE id = 3;

COMMIT;

JDBC — Java'dan SQL'ga

JDBC (Java Database Connectivity) — Java'ning standart database API'si. 

String url = "jdbc:postgresql://localhost:5432/mydb";
String user = "admin";
String password = "secret";

try (Connection conn = DriverManager.getConnection(url, user, password)) {

    // SELECT
    String sql = "SELECT id, name, email FROM users WHERE age > ?";
    try (PreparedStatement ps = conn.prepareStatement(sql)) {
        ps.setInt(1, 18);

        try (ResultSet rs = ps.executeQuery()) {
            while (rs.next()) {
                long id = rs.getLong("id");
                String name = rs.getString("name");
                System.out.println(id + " " + name);
            }
        }
    }

    // INSERT
    String insert = "INSERT INTO users (name, email) VALUES (?, ?)";
    try (PreparedStatement ps = conn.prepareStatement(insert)) {
        ps.setString(1, "Akobir");
        ps.setString(2, "akobir@mail.com");
        int rows = ps.executeUpdate();
    }
}

SQL Injection — xavf va himoya

SQL Injection — eng xavfli web zaifliklardan biri. Foydalanuvchi inputi to'g'ridan-to'g'ri SQL'ga qo'shilsa, hacker o'z so'rovini bajartirishi mumkin. 
// XAVFLI — String concatenation
String query = "SELECT * FROM users WHERE name = '" + userName + "'";

// Hacker shunday yozadi:
// userName = "x'; DROP TABLE users; --"
// Natija: SELECT * FROM users WHERE name = 'x'; DROP TABLE users; --'
// USERS JADVALI YO'Q BO'LADI!

Yechim — PreparedStatement (parameterized query):

SQL strukturasi va parametr alohida uzatiladi. Parametr har doim qiymat sifatida tushuniladi, hech qachon SQL kodi sifatida emas.

// XAVFSIZ — PreparedStatement
String query = "SELECT * FROM users WHERE name = ?";
PreparedStatement ps = conn.prepareStatement(query);
ps.setString(1, userName);  // har qanday input — xavfsiz!
ResultSet rs = ps.executeQuery();
Spring Data JPA ham PreparedStatement'dan foydalanadi — shuning uchun xavfsiz.
📝 Amaliy topshiriqlar Mustaqil

10 ta murakkab vazifa

Quyidagi vazifalar uchun PostgreSQL'da o'zingiz jadval yarating va SQL yozing:

  • 1. E-commerce schema. users, products, orders, order_items, categories jadvallari. Foreign key'lar bilan.
  • 2. Eng ko'p sotuvchi shahar. Mahsulotlar narxi va sotilgan miqdoriga ko'ra eng ko'p daromad keltirgan shaharni toping.
  • 3. Customer Lifetime Value. Har bir mijozning umumiy xarid summasi va birinchi/oxirgi xarid sanasi.
  • 4. Retention analysis. Birinchi marta xarid qilgan va 30 kun ichida yana qaytgan mijozlar ulushi.
  • 5. Window functions. Har bir shaharda balans bo'yicha top-3 foydalanuvchini ROW_NUMBER() bilan.
  • 6. Recursive query. Kategoriyalar daraxti (parent_id bilan). Berilgan kategoriya va uning barcha child'larini toping (CTE).
  • 7. Aralashtirilgan transaction. Pul o'tkazmasi: tekshirish, yechish, qo'shish, log yozish — hammasi atomik.
  • 8. Index optimization. Slow query'ni EXPLAIN ANALYZE bilan tahlil qiling va index qo'shing.
  • 9. Pivot table. Oylik daromadni har bir kategoriya bo'yicha pivot qiling (CASE WHEN + SUM).
  • 10. JDBC + Connection Pool. HikariCP bilan connection pool sozlang. Batch insert (10000 ta qator) qiling.
