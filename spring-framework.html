Spring Framework

Java backend dasturchining asosiy quroli. IoC, Dependency Injection, Bean'lar va kuchli ekosistema.

Dars maqsadi: Spring'ning falsafasi (IoC, DI), Bean lifecycle, asosiy annotatsiyalar (@Component, @Service, @Repository, @Controller, @Autowired), AOP va konfiguratsiyalarni chuqur o'rganamiz.

Spring nima va nima uchun?

Spring — Java uchun eng mashhur va kuchli enterprise framework. 2003-yilda Rod Johnson tomonidan yaratilgan.

Asosiy falsafasi: Dasturchi "WHAT"ni yozadi, Spring "HOW"ni hal qiladi.

Boilerplate kodni kamaytirish, modulli arxitektura va sinab ko'rishni osonlashtirish.

  • Lightweight — ortiqcha narsa yo'q, kerakli moduli olasiz
  • Modulli — Core, MVC, Data, Security, Cloud va h.k.
  • Testlash oson — har bir komponent mustaqil sinaladi
  • POJO-asosida — oddiy Java obyektlar bilan ishlaydi
  • Ko'p integratsiya — JPA, Hibernate, Kafka, Redis va h.k.

Spring ekosistemasi

Spring Core

IoC container, DI, Bean lifecycle — barchasining negizi.

Spring MVC

Web ilovalar uchun Model-View-Controller framework.

Spring Data

Database bilan ishlash — JPA, MongoDB, Redis, JDBC.

Spring Security

Auth, authorization, JWT, OAuth2 — xavfsizlik.

Spring Cloud

Microservice'lar — Config Server, Discovery, Gateway.

Spring Boot

Hamma narsani tayyor sozlash bilan birga olib keluvchi qatlam.

IoC — Inversion of Control

An'anaviy yondashuv: Dastur o'zi obyektlarni yaratadi, ularni bog'laydi, hayot davrini boshqaradi.

IoC bilan: Bu vazifani framework (container) ga topshiramiz. Biz faqat e'lon qilamiz: "menga shu obyekt kerak". Spring uni yaratib beradi.

"Don't call us, we'll call you" — Hollywood Principle.

// IoC SIZ — o'zimiz boshqaramiz
public class OrderService {
    private final UserRepository userRepo;
    private final EmailService email;

    public OrderService() {
        // Qattiq bog'liqlik (tight coupling)
        this.userRepo = new UserRepository();
        this.email = new EmailService();
    }
}

// Muammolar:
// - Boshqa UserRepository qo'yib bo'lmaydi
// - Mock qilib test yozish qiyin
// - Bog'liqliklar yashirin
// IoC bilan — Spring boshqaradi
@Service
public class OrderService {
    private final UserRepository userRepo;
    private final EmailService email;

    public OrderService(UserRepository userRepo, EmailService email) {
        this.userRepo = userRepo;
        this.email = email;  // Spring beradi!
    }
}

DI — Dependency Injection

DI — IoC ning amaliy ifodasi. Bog'liqliklarni tashqaridan berish.

DI 3 ta usulda ishlaydi: Constructor, Setter, Field.

Constructor Injection — eng tavsiya etilgan: final qilib bo'ladi, mock qilish oson, NPE'ga olib kelmaydi.

// 1. CONSTRUCTOR INJECTION (TAVSIYA)
@Service
public class UserService {
    private final UserRepository repo;

    public UserService(UserRepository repo) {  // @Autowired ixtiyoriy
        this.repo = repo;
    }
}

// 2. SETTER INJECTION (kam ishlatiladi)
@Service
public class UserService {
    private UserRepository repo;

    @Autowired
    public void setRepo(UserRepository repo) {
        this.repo = repo;
    }
}

// 3. FIELD INJECTION (TAVSIYA ETILMAYDI)
@Service
public class UserService {
    @Autowired
    private UserRepository repo;  // final emas, mock qiyin
}
Lombok bilan: @RequiredArgsConstructor + final field — eng qulay yo'l!

ApplicationContext va Bean

ApplicationContext — Spring'ning IoC containeri. Bean'larni yaratadi, ularni boshqaradi.

Bean — Spring boshqaruvidagi obyekt. Spring uni yaratib, lifecycle'ini olib boradi, kerakli joyga inject qiladi.

// Bean yaratish — bir nechta yo'l

public class MyService { /* ... */ }

// 2. @Bean metod konfiguratsiyada
@Configuration

    @Bean

        return new MyService();
    }
}

// 3. XML (eski usul, hozir kam)

// Bean'ni olish (odatda DI orqali, qo'lda kam)
ApplicationContext ctx = SpringApplication.run(App.class, args);
MyService service = ctx.getBean(MyService.class);

Stereotype annotatsiyalar

Spring'da @Component'ning maxsus versiyalari bor — har biri biznes mantiqi qatlami uchun.

@Component

Umumiy bean. Boshqalarining "ota"si.

@Service

Biznes mantiqi qatlami (Service layer).

@Repository

Ma'lumotlar bilan ishlash (DAO). DB exception'larini Spring exception'lariga aylantiradi.

@Controller

Web qatlam (MVC). HTTP so'rovlarni qabul qiladi.

Texnik jihatdan ular bir xil — barchasi @Component. Lekin semantik jihatdan kodning maqsadini ko'rsatadi (clean architecture).

Strategy Pattern + Bean'lar

PaymentStrategy interfeysi bor: CashPayment, CardPayment, CryptoPayment bean'lari. Foydalanuvchi tanlagan turdagi bean'ni dynamic tarzda oling.

Misol
pay("CARD", 100) → CardPayment.process(100)
pay("CRYPTO", 100) → CryptoPayment.process(100) 
public interface PaymentStrategy {
    String type();
    void process(double amount);
}

@Component
public class CardPayment implements PaymentStrategy {
    public String type() { return "CARD"; }
    public void process(double amount) { /* ... */ }
}

@Component
public class CashPayment implements PaymentStrategy {
    public String type() { return "CASH"; }
    public void process(double amount) { /* ... */ }
}

@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> strategies;

    public PaymentService(List<PaymentStrategy> list) {
        this.strategies = list.stream()
            .collect(Collectors.toMap(PaymentStrategy::type, s -> s));
    }

    public void pay(String type, double amount) {
        PaymentStrategy s = strategies.get(type);
        if (s == null) throw new IllegalArgumentException("Unknown: " + type);
        s.process(amount);
    }
}

Bean Lifecycle

Spring'da har bir bean quyidagi hayot bosqichlaridan o'tadi:

1. Container ishga tushadi
   ↓
2. Bean instantsiyalanadi (constructor)
   ↓
3. Dependency injection (Setter, Field)
   ↓
4. @PostConstruct chaqiriladi
   ↓
5. InitializingBean.afterPropertiesSet() (agar implementatsiya qilingan bo'lsa)
   ↓
6. Custom init metod (agar @Bean(initMethod="..."))
   ↓
7. ✅ BEAN TAYYOR — foydalanish mumkin
   ↓
8. Container yopilganda:
   - @PreDestroy chaqiriladi
   - DisposableBean.destroy()
   - Custom destroy metod
@Component
public class MyBean {
    public MyBean() {
        System.out.println("1. Constructor");
    }

    @PostConstruct
    public void init() {
        System.out.println("2. PostConstruct — bean tayyor");
        // resurslarni ochish, cache to'ldirish, va h.k.
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("3. PreDestroy — yopilmoqda");
        // resurslarni yopish
    }
}

@Conditional + Custom Annotation

Custom annotation @FeatureFlag("dark-mode") yarating. Faqat features.dark-mode=true bo'lsa bean yaratilsin. 

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Conditional(FeatureFlagCondition.class)
public @interface FeatureFlag {
    String value();
}

public class FeatureFlagCondition implements Condition {
    @Override
    public boolean matches(ConditionContext ctx,
                          AnnotatedTypeMetadata meta) {
        Map<String, Object> attrs = meta.getAnnotationAttributes(
            FeatureFlag.class.getName());
        String flagName = (String) attrs.get("value");

        String key = "features." + flagName;
        return "true".equals(ctx.getEnvironment().getProperty(key));
    }
}

@Service
@FeatureFlag("dark-mode")
public class DarkModeService {
    // faqat features.dark-mode=true bo'lganda yaratiladi
}

AOP — Aspect-Oriented Programming

Muammo: Logging, transaction, security, cache — bu kod har joyda takrorlanadi (cross-cutting concerns).

Yechim: AOP — bu kodni alohida joyga (Aspect) chiqarib, kerakli metodlarga avtomatik "yopishtirish".

AtamaTavsif
AspectCross-cutting concern (logging, sec)
Join PointMetod chaqiruv nuqtasi
PointcutQaysi metodlar uchun (naqsh)
AdviceNima qilish (before, after, around)
WeavingKodga aspect'ni bog'lash jarayoni

AOP Annotatsiyalari

@Aspect
@Component
public class LoggingAspect {

    // Pointcut — qaysi metodlar
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() {}

    // BEFORE — metoddan oldin
    @Before("serviceMethods()")
    public void logBefore(JoinPoint jp) {
        System.out.println("➡ " + jp.getSignature().getName());
    }

    // AFTER — metoddan keyin (har doim)
    @After("serviceMethods()")
    public void logAfter(JoinPoint jp) {
        System.out.println("⬅ " + jp.getSignature().getName());
    }

    // AROUND — eng kuchli, hammasini boshqaradi
    @Around("serviceMethods()")
    public Object measure(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed();  // asl metodni chaqirish
            return result;
        } finally {
            long time = System.currentTimeMillis() - start;
            System.out.println("⏱ " + pjp.getSignature().getName() + ": " + time + "ms");
        }
    }

    // AFTER RETURNING — faqat muvaffaqiyatli tugaganda
    @AfterReturning(pointcut = "serviceMethods()", returning = "result")
    public void logReturn(Object result) {
        System.out.println("Result: " + result);
    }

    // AFTER THROWING — exception bo'lganda
    @AfterThrowing(pointcut = "serviceMethods()", throwing = "ex")
    public void logError(Exception ex) {
        System.out.println("Error: " + ex.getMessage());
    }
}

Cache aspect

Custom @Cacheable aspect yarating — metod natijasini Map'da saqlasin va keyingi safarlarda qaytarsin. 

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface SimpleCache { }

@Aspect
@Component
public class SimpleCacheAspect {
    private final Map<String, Object> cache = new ConcurrentHashMap<>();

    @Around("@annotation(SimpleCache)")
    public Object cacheResult(ProceedingJoinPoint pjp) throws Throwable {
        String key = pjp.getSignature().toString() + Arrays.toString(pjp.getArgs());

        if (cache.containsKey(key)) {
            System.out.println("HIT: " + key);
            return cache.get(key);
        }

        System.out.println("MISS: " + key);
        Object result = pjp.proceed();
        cache.put(key, result);
        return result;
    }
}

@Service
public class ProductService {
    @SimpleCache
    public Product findById(Long id) {
        // sekin operatsiya
        return new Product(id, "Item");
    }
}

📝 Amaliy topshiriqlar Mustaqil

10 ta murakkab vazifa

Quyidagi vazifalarni Spring Boot loyihasi yaratib bajaring:

  • 1. To'liq Service-Repository chain. ProductController → ProductService → ProductRepository. findAll, findById, save, delete.
  • 2. Constructor injection + Lombok. @RequiredArgsConstructor va final field'lar bilan.
  • 3. Multi-implementation Strategy. 3 ta Notifier (Email, SMS, Push) — List<Notifier> orqali aniqlash.
  • 4. @Profile dev vs prod. Dev'da console'ga log, prod'da real serverga jo'natish.
  • 5. @ConfigurationProperties. Murakkab konfiguratsiyani tip-safe class'ga to'plang (nested obyektlar bilan).
  • 6. @PostConstruct cache loader. Application ishga tushganda DB'dan ma'lumotni o'qib, cache'ga to'ldirish.
  • 7. @Async metod. Email yuborishni async qiling. @EnableAsync + @Async annotatsiyalari.
  • 8. AOP Audit Aspect. Hamma @Repository metodlari uchun audit log (kim, qachon, qaysi argumentlar bilan).
  • 9. Custom Validation. @ValidPhone annotatsiya yarating — Uzbekiston raqamlarini tekshiruvchi ConstraintValidator.
  • 10. Event-driven communication. ApplicationEventPublisher bilan UserRegisteredEvent — bir nechta listener (email, log, metric) reaksiya qilsin.
Har bir vazifa uchun: SOLID tamoyillariga rioya qiling, final field'lar ishlatib bog'liqliklarni aniq ko'rsating, Constructor Injection bilan yozing.
