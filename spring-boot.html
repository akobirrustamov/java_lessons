Spring Boot

Spring Framework ustiga qurilgan opinionated qatlam. Auto-configuration, starter'lar va embedded server bilan — daqiqada production-ready ilova.

Dars maqsadi: Auto-configuration, starter dependencies, application.properties/yml, profiles, logging, validation va exception handling'ni nazariya va amaliyot bilan chuqur o'rganamiz.

Spring Boot nima?

Spring Boot — 2014-yilda chiqarilgan, Spring'ni oddiy va tez ishlatish uchun yaratilgan framework.

Asosiy g'oya: "Convention over Configuration" — har bir konfiguratsiyani qo'lda yozish o'rniga, oqilona standartlarni belgilash.

Spring Boot 4 ta asosiy printsipga tayanadi:

  • Auto-configuration — kerakli konfiguratsiya avtomatik qo'shiladi
  • Starter dependencies — bog'liqliklarning tematik to'plamlari
  • Embedded server — Tomcat/Jetty/Undertow JAR ichida
  • Production-ready — Actuator, metrics, health checks tayyor
Natija: java -jar app.jar bilan ishga tushadigan standalone ilova. Tashqi server o'rnatish shart emas.

Auto-configuration — ichki ishlash

Auto-configuration nima? Classpath'da qaysi kutubxonalar borligini ko'rib, Spring kerakli bean'larni avtomatik yaratadi.

Misol: classpath'da spring-boot-starter-web bo'lsa → Tomcat + DispatcherServlet + Jackson tayyor.

Asosiy mexanizm: META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports faylida ro'yxat. Har bir konfiguratsiya @Conditional... orqali "qachon ishga tushishi"ni belgilaydi.

// Spring Boot'ning ichki konfiguratsiyasi (soddalashtirilgan)
@AutoConfiguration
@ConditionalOnClass(DataSource.class)        // DataSource klassi bo'lsa
@ConditionalOnMissingBean(DataSource.class)  // bizning DataSource yo'q bo'lsa
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {

    @Bean
    public DataSource dataSource(DataSourceProperties props) {
        return DataSourceBuilder.create()
            .url(props.getUrl())
            .username(props.getUsername())
            .build();
    }
}
Auto-config'ni o'chirish: @SpringBootApplication(exclude = DataSourceAutoConfiguration.class) yoki spring.autoconfigure.exclude property.

Property hierarchy — qaysi qiymat qo'llaniladi?

1. Command-line argumentlar           ← eng yuqori prioritet

2. SPRING_APPLICATION_JSON
   { "server": { "port": 9000 } }

3. JNDI atributlari (java:comp/env)

4. Java System properties
   -Dserver.port=9000

5. OS Environment variables
   SERVER_PORT=9000 java -jar app.jar

6. application-{profile}.yml (active profile)
7. application-{profile}.properties

8. application.yml / application.properties  ← eng past prioritet

9. @PropertySource annotatsiya bilan
Amaliyot: Production'da environment variables ishlatish tavsiya etiladi (12-factor app printsipi). Parol va sirlarni hech qachon kod ichiga yozmang!

Multi-profile EmailService

EmailService interfeysi: dev'da console'ga yozadi, prod'da haqiqiy SMTP'ga jo'natadi. Profile orqali tanlanadi. 

public interface EmailService {
    void send(String to, String subject, String body);
}

@Service
@Profile("dev")
@Slf4j
public class ConsoleEmailService implements EmailService {
    @Override
    public void send(String to, String subject, String body) {
        log.info("FAKE EMAIL → {}: [{}] {}", to, subject, body);
    }
}

@Service
@Profile("prod")
public class SmtpEmailService implements EmailService {
    private final JavaMailSender mailSender;

    public SmtpEmailService(JavaMailSender mailSender) {
        this.mailSender = mailSender;
    }

    @Override
    public void send(String to, String subject, String body) {
        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setTo(to);
        msg.setSubject(subject);
        msg.setText(body);
        mailSender.send(msg);
    }
}

@Service

public class ProductService {
    private final ProductRepository repo;
    private final AtomicInteger requestCount = new AtomicInteger(0);

    public ProductService(ProductRepository repo) { this.repo = repo; }

    public Product create(Product p) {
        int count = requestCount.incrementAndGet();
        if (count > 100) {
            log.warn("High request rate: {} requests", count);
        }

        try {
            Product saved = repo.save(p);
            log.info("Product created: id={}, name={}", saved.getId(), saved.getName());
            return saved;
        } catch (Exception e) {
            log.error("Failed to create product: {}", p.getName(), e);
            throw e;
        }
    }
}

Request ID bilan log (MDC)

Har bir HTTP request uchun noyob ID generate qiling va u barcha log qatorlariga avtomatik qo'shilsin. 

@Component
@Order(1)
public class RequestIdFilter extends OncePerRequestFilter {
    public static final String REQ_ID = "requestId";

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                   HttpServletResponse res,
                                   FilterChain chain)
            throws ServletException, IOException {
        try {
            String reqId = UUID.randomUUID().toString().substring(0, 8);
            MDC.put(REQ_ID, reqId);          // MDC — Mapped Diagnostic Context
            res.setHeader("X-Request-Id", reqId);
            chain.doFilter(req, res);
        } finally {
            MDC.clear();
        }
    }
}
logging:
  pattern:
    console: "%d{HH:mm:ss} %-5level [%X{requestId}] %logger - %msg%n"
    # %X{requestId} — MDC'dan oladi

Bean Validation — JSR-380

Bean Validation — Java EE standarti (JSR-380). Obyekt field'lariga annotatsiya qo'yish orqali validatsiya qoidalarini belgilash.

Hibernate Validator — bu standartning eng mashhur implementatsiyasi. Spring Boot'da spring-boot-starter-validation bilan keladi.

Foyda: Validatsiya kodi — annotatsiya bo'lib obyekt ichida turadi. DRY principle.

AnnotationTavsif
@NotNullnull bo'lmasin
@NotBlankString null/bo'sh/probelli bo'lmasin
@NotEmptyString yoki collection bo'sh emas
@Size(min, max)String/collection o'lchami
@Min, @MaxSonning chegarasi
@EmailEmail format
@Pattern(regexp)Regex bilan tekshirish
@Positive, @NegativeMusbat/manfiy son
@Past, @FutureSanaga oid

@RestControllerAdvice
public class ValidationHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Map<String, Object> handle(MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();

        ex.getBindingResult().getFieldErrors().forEach(err ->
            errors.put(err.getField(), err.getDefaultMessage())
        );

        return Map.of(
            "timestamp", LocalDateTime.now(),
            "status", 400,
            "errors", errors
        );
    }
}

Custom validator — UzPhone

@UzPhone annotatsiya yarating: +998XXXXXXXXX formatini tekshirsin (9 ta raqam). 

@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = UzPhoneValidator.class)
public @interface UzPhone {
    String message() default "Telefon formati: +998XXXXXXXXX";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class UzPhoneValidator implements ConstraintValidator<UzPhone, String> {

    private static final Pattern PATTERN = Pattern.compile("^\\+998\\d{9}$");

    @Override
    public boolean isValid(String value, ConstraintValidatorContext ctx) {
        if (value == null) return true;  // null bo'lsa — boshqa @NotNull tekshiradi
        return PATTERN.matcher(value).matches();
    }
}

// Foydalanish
public class UserRequest {
    @UzPhone
    private String phone;
}

Exception Handling — local va global

// 1. LOCAL — bir Controller ichida
@RestController
public class UserController {

    @ExceptionHandler(UserNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(UserNotFoundException ex) {
        return new ErrorResponse("USER_NOT_FOUND", ex.getMessage());
    }

    @GetMapping("/users/{id}")
    public User get(@PathVariable Long id) {
        return service.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
    }
}
// 2. GLOBAL — barcha controller'lar uchun
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(EntityNotFoundException ex) {
        log.warn("Entity not found: {}", ex.getMessage());
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(new ErrorResponse("NOT_FOUND", ex.getMessage()));
    }

    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<ErrorResponse> handleForbidden(AccessDeniedException ex) {
        return ResponseEntity.status(HttpStatus.FORBIDDEN)
            .body(new ErrorResponse("FORBIDDEN", "Ruxsat yo'q"));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(Exception ex) {
        log.error("Unexpected error", ex);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(new ErrorResponse("INTERNAL_ERROR", "Server xatosi"));
    }
}

ProblemDetail — RFC 7807

ProblemDetail (Spring 6+) — HTTP API xatolari uchun standart format. RFC 7807 standarti.

JSON yoki XML formatida bir xil ko'rinishdagi xatolar. Frontend uchun parse qilish oson.

@RestControllerAdvice
public class ProblemHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ProblemDetail handleNotFound(UserNotFoundException ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.NOT_FOUND);
        problem.setTitle("User Not Found");
        problem.setDetail(ex.getMessage());
        problem.setType(URI.create("https://api.example.com/errors/user-not-found"));
        problem.setProperty("userId", ex.getUserId());
        problem.setProperty("timestamp", Instant.now());
        return problem;
    }
}

10 ta murakkab vazifa

Quyidagi vazifalarni Spring Boot loyihasida bajaring:

  • 1. Multi-profile loyiha. dev (H2), test (Testcontainers), prod (PostgreSQL) — har biri uchun alohida config.
  • 2. @ConfigurationProperties nested. Murakkab konfiguratsiyani type-safe class'larga ajrating (nested obyektlar bilan).
  • 3. Custom validator group'lar bilan. Bir DTO uchun: create'da hamma field majburiy, update'da ixtiyoriy.
  • 4. Global Exception Handler. 10+ exception turi uchun ProblemDetail formatda javob.
  • 5. MDC bilan log enrichment. Request ID, user ID, IP — har log qatorga avtomatik qo'shilsin.
  • 6. Async + log context. @Async metodda parent thread'ning MDC'sini saqlash (TaskDecorator).
  • 7. Conditional bean. app.features.cache=true bo'lsa Redis cache, aks holda in-memory cache.
  • 8. Custom Actuator endpoint. /actuator/cache-stats — cache hit/miss ratio'sini ko'rsatsin.
  • 9. Integration test with Testcontainers. PostgreSQL'ni Docker container'da ko'tarib, real test yozish.
  • 10. Production-ready config. Secrets externalization, structured logging, health checks, graceful shutdown.
Har bir vazifa uchun: 12-factor app tamoyillariga rioya qiling, secrets'ni hech qachon kod ichiga yozmang.
